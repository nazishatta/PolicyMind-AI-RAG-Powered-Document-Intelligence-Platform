# PolicyMind-AI — Project Summary for Recruiters and Technical Reviewers

> **Audience:** Recruiters, hiring managers, technical reviewers, AI/ML engineers, backend engineers, open-source reviewers.
> **Last verified:** Test suite passes at 405 tests (1 skipped). All code references were confirmed by reading source files before writing this document.

## 1. Executive Summary

PolicyMind-AI is a backend API platform that applies GraphRAG (Graph-augmented Retrieval-Augmented Generation) to the problem of extracting structured, citation-backed answers from policy documents. The system ingests plain-text or publicly accessible PDF documents, builds a knowledge graph from named entities and their relationships, and answers natural-language questions using a hybrid retrieval pipeline that fuses semantic vector similarity with graph-neighbourhood evidence. Every answer carries a structured citation list, a numerical confidence score, a human-readable confidence note, an evidence-quality label, and a deterministic list of limitations — all computed without trusting the LLM to self-report its own uncertainty.

The project demonstrates professional-grade backend engineering: an API-first architecture built on FastAPI and Pydantic v2, a modular service layer where every component is independently testable, a 405-test suite that runs fully offline, three-job GitHub Actions CI (lint, multi-version unit tests, and an end-to-end smoke test), and GitHub Codespaces support for zero-setup evaluation. It is intentionally designed as a portfolio project that can be verified, extended, and demonstrated by any reviewer without requiring API keys, database installs, or large dataset downloads.

## 2. Project Purpose

PolicyMind-AI was built to address a specific class of knowledge-work problem: analysing long, structured public-interest documents — policy frameworks, legislation, international agreements, public-health guidance — where the quality of an answer depends not just on finding similar text but on understanding the named entities, institutions, and relationships those documents contain.

The platform serves four intersecting purposes:

**Document intelligence.** Policy documents are not homogeneous prose. They have article structures, numerical targets, named agencies, legal deadlines, and cross-references. PolicyMind-AI captures this structure through section-heading detection during chunking, spaCy-based named entity recognition during ingestion, and rule-based relation extraction that produces a lightweight knowledge graph alongside the vector index.

**Citation-backed question answering.** Every answer is grounded in retrieved source passages. The system prompt enforces inline page citations before and after each factual claim. The API response includes a `citations` list where every entry carries `chunk_id`, `doc_id`, `doc_title`, `page_number`, `section_heading`, `excerpt`, and `relevance_score` — enough to trace any claim to its exact location in the source document.

**GraphRAG hybrid retrieval.** Beyond vector similarity, the retriever walks the knowledge graph to find chunks whose entities have well-connected graph evidence. This means a chunk about the "European Commission" ranks higher when graph edges confirm that entity's policy role, even if pure cosine similarity is modest.

**Trust-aware AI output.** Every response includes `answer_type` (cited / partial / refused / no_corpus), `evidence_quality` (strong / moderate / weak / insufficient), `confidence` (float [0,1]), `confidence_note` (human-readable explanation), and `limitations` (deterministic list of caveats). These fields are computed by the pipeline, not generated by the LLM, making them reliable regardless of which LLM provider is active.

## 3. Problem Statement

Standard approaches to "chat with documents" fail in several specific ways that this project addresses directly:

**Long documents are hard to search.** A 200-page policy framework cannot be loaded into a context window. Retrieval must identify the relevant pages from among thousands of chunks before any LLM is involved.

**Generic chat-PDF tools lack source tracing.** Most consumer tools return an answer with no indication of which passage it came from, which page, or what the confidence is. This is unacceptable for policy research, journalism, or any application where a claim might be acted upon.

**Standard vector search misses entity relationships.** If a user asks "which agencies enforce the just transition obligations?", pure cosine similarity retrieves passages that are textually similar to the question. But the answer might depend on knowing that "Ministry of Environment" and "Regional Development Authority" are both categorised as enforcement bodies in the graph — information that is explicit in the document structure but diffuse in raw text.

**Policy documents require explicit confidence and limitations.** A political scientist or policy analyst needs to know not just the answer but how confident the system is and what it cannot answer. Systems that produce confident-sounding text with no uncertainty signal are actively dangerous for high-stakes research use cases. The trust layer in this project makes uncertainty explicit and machine-readable.

## 4. Solution Overview

PolicyMind-AI solves these problems through a six-stage pipeline:

1. **Ingest.** Accept a document as a raw text payload or a publicly accessible URL (PDF or plain text). Parse and normalise into a `PolicyDocument` object with page-level structure. Detect section headings.

2. **Chunk and embed.** Split the document into overlapping `TextChunk` objects of ~512 characters with a 64-character overlap. Each chunk carries its `doc_id`, `page_number`, `section_heading`, `chunk_id`, and character offsets. Generate 384-dimensional dense embeddings using the `sentence-transformers/all-MiniLM-L6-v2` model. Store in ChromaDB (persistent) or in-memory.

3. **Extract and graph.** Run spaCy NER (with regex fallback when spaCy is absent) to extract policy-relevant entities: organisations, geopolitical entities, legislation, money, percentages, dates, products, events, and nationality groups. Apply rule-based co-occurrence relation extraction to produce typed triples: `MENTIONED_WITH`, `OPERATES_IN`, `TARGETS_BY`, `GOVERNED_BY`. Store in a NetworkX directed graph (or Neo4j when configured).

4. **Retrieve.** On query, embed the question and fetch 2× the requested number of candidate chunks by cosine similarity. Walk the knowledge graph for matched entities up to `graph_depth` hops (1–3, default 1). Apply proportional score fusion: `fused = 0.7 × vector_score + 0.3 × graph_boost`, where `graph_boost = Σ entity_confidences / |entity_set|`.

5. **Generate with citations.** If no chunks are retrieved, return a structured refusal without calling the LLM. Otherwise, build a context block with page-number markers and invoke the LLM (Anthropic Claude, OpenAI GPT-4, or Mock). The system prompt mandates citation-first structure and inline page references. After generation, rerank citations so pages explicitly named in the answer appear first.

6. **Apply trust layer.** Compute confidence score, confidence note, evidence-quality label, answer-type classification, and limitations list. Return all fields in a structured `AnswerResponse`.

## 5. Step-by-Step System Workflow

The following numbered steps trace a complete request from document submission to structured response:

1. **User submits a document** via `POST /api/v1/ingest` with either a `text` field (raw content) or a `url` field (publicly accessible PDF or text URL) plus an optional title and source label.

2. **API validates the request.** Pydantic v2 validates the `IngestRequest` schema. The route handler calls `connector_from_request()` to select the correct connector (text or URL). Typed `IngestionError` subclasses map to appropriate HTTP status codes: 413 for oversized files, 415 for unsupported types, 502 for fetch failures, 422 for empty or malformed content.

3. **`document_loader` processes the content.** For URL ingestion, `load_from_url()` streams the document over httpx with configurable timeouts and a hard file-size cap. For PDFs, it tries `pdfplumber` first (higher fidelity) and falls back to `PyPDF2`. For text, it normalises encoding and whitespace. The result is a `PolicyDocument` dataclass with a stable `doc_id` (SHA-256 of normalised title + text), `pages: list[tuple[int, str]]`, `page_count`, and `source_url`.

4. **`text_chunker` creates citation-aware chunks.** The recursive semantic chunker splits text along a separator hierarchy (paragraph breaks → sentences → hard character limit). Each `TextChunk` carries `chunk_id`, `doc_id`, `page_number`, `section_heading` (detected from the first five lines of the chunk using a regex covering article/section/numbered/ALL-CAPS patterns), `char_start`, `char_end`, and `text`.

5. **The embedding layer generates vectors.** `EmbeddingModel.embed()` calls `SentenceTransformer.encode()` with `normalize_embeddings=True`, producing normalised float32 vectors of dimension 384. The model is lazy-loaded and process-level cached to avoid reload overhead across requests.

6. **The vector store indexes the chunks.** `BaseVectorStore.upsert()` stores chunks and embeddings together with their metadata. ChromaDB is the default persistent store; `InMemoryVectorStore` (NumPy dot-product similarity) is used in tests and mock mode to avoid the ChromaDB dependency.

7. **Entity extraction builds named entity data.** `extract_entities()` calls `spacy.load("en_core_web_sm")` and filters to the nine policy-relevant entity types. When spaCy is absent, the system logs a debug message and returns an empty list — ingestion continues without graph data. `extract_relations()` applies proximity rules over sorted entity pairs to generate typed triples.

8. **The graph service stores the graph data.** `InMemoryGraphService` uses a NetworkX `DiGraph` where nodes are entity names and edges carry `source`, `relation`, `target`, `doc_id`, and `confidence`. The optional Neo4j adapter satisfies the same `BaseGraphService` interface.

9. **The query endpoint receives a question.** `POST /api/v1/query` accepts a `QueryRequest` with `question` (5–2000 chars), optional `doc_id` filter, optional `top_k` override (1–20), `include_graph_evidence` (bool), and optional `graph_depth` (1–3). All validated by Pydantic v2 before the handler is called.

10. **The retriever performs semantic search.** `HybridRetriever.retrieve()` embeds the question, fetches `top_k × 2` candidates by cosine similarity (capped at `top_k + 10`), and applies an optional `doc_id` filter.

11. **The retriever expands the graph neighbourhood.** `_expand_graph()` extracts entity names from candidate chunk texts (spaCy or regex), queries the graph for direct edges, then for `depth > 1` collects neighbours' edges with confidence discounted by `_MULTIHOP_DISCOUNT = 0.8` per hop. Triples are deduplicated across the full multi-hop pass. Entity confidence scores (mean edge confidence per entity) are returned for use in score fusion.

12. **Score fusion reranks candidates.** `_fuse_scores()` computes `fused = α × vector_score + (1 − α) × graph_boost` for each candidate, where `graph_boost = Σ entity_confidences[e] / |entity_set|` summed over matched entities. Chunks covering more of the query's relevant entities at higher graph confidence rank higher. The top `top_k` candidates are returned.

13. **The citation engine records sources.** `CitationEngine.record()` accumulates each retrieved chunk's provenance fields. After LLM generation, `_rerank_citations_by_answer()` parses page-number references from the answer text using the regex `\bp\.?\s*(\d+)\b` and stable-sorts citations so explicitly referenced pages appear first.

14. **The LLM or mock provider generates the answer.** The system prompt mandates citation-first structure, inline page references, refusal when unsupported, and a `"Based on limited evidence:"` prefix for partial answers. When confidence < 0.35, a retrieval note is prepended to the user prompt instructing the LLM to qualify its response. `BaseLLMProvider.acomplete()` is called asynchronously; the mock provider returns a structured stub answer reflecting the actual context without any API call.

15. **The trust layer computes all metadata.** Five functions produce the trust fields: `_compute_confidence()` (mean top-3 cosine + graph bonus, capped at 1.0), `_compute_confidence_note()` (human-readable breakdown), `_classify_answer_type()` (cited / partial / refused / no_corpus), `_classify_evidence_quality()` (strong / moderate / weak / insufficient), and `_infer_limitations()` (deterministic list of active caveats). None of these call the LLM.

16. **The API returns a structured `AnswerResponse`.** The response carries all 14 fields: `query_id`, `question`, `answer`, `answer_type`, `evidence_quality`, `confidence_note`, `citations`, `retrieved_chunks`, `graph_evidence`, `confidence`, `limitations`, `latency_ms`, `provider`, and `model`. End-to-end latency is measured with `time.perf_counter()` and rounded to 0.1 ms.

## 9. API Platform

The API is documented interactively at `/docs` (Swagger UI) and `/redoc` (ReDoc) when the server is running. Full request/response examples are in `docs/api_usage.md` and `examples/sample_api_payloads.json`.

### `GET /`
**Purpose:** Root index — lists all available endpoints and active provider configuration.
**Input:** None.
**Output:** `RootResponse` with `name`, `version`, `description`, `status`, `docs_url`, `redoc_url`, `endpoints` (dict), `providers` (dict).
**Why it matters:** Allows any consumer to discover available endpoints without reading documentation. The `providers` dict confirms which LLM, vector store, and graph backend are active — useful for debugging deployment configuration.

---

### `GET /health`
**Purpose:** Enriched health check for monitoring and CI readiness polling.
**Input:** None.
**Output:** `HealthResponse` with `status`, `version`, `llm_provider`, `llm_model`, `vector_store`, `vector_store_chunks` (int), `graph_provider`, `graph_enabled` (bool), `graph_entities` (int), `graph_relations` (int).
**Why it matters:** Provides operational observability. The CI smoke test polls this endpoint until the server is ready before running further checks. The chunk and entity counts confirm that ingestion is working without needing to call the query endpoint.

---

### `POST /api/v1/ingest`
**Purpose:** Ingest a policy document into the retrieval index.
**Input:** `IngestRequest` — either `text: str` (raw content) or `url: HttpUrl` (publicly accessible PDF or text); optional `title`, `source_label`, `max_file_size_mb`.
**Output:** `IngestResponse` — `doc_id` (stable SHA-256 hash), `title`, `source_url`, `page_count`, `chunk_count`, `entities_extracted`, `graph_nodes_added`, `processing_status`, `message`.
**Why it matters:** The entry point for all document intelligence. Re-ingesting the same document (same title + text) returns the same `doc_id` — idempotent by design. Typed error hierarchy maps failures to correct HTTP codes (413, 415, 502, 422) so consumers can handle each case programmatically.

---

### `POST /api/v1/query`
**Purpose:** Answer a natural-language question over the indexed corpus.
**Input:** `QueryRequest` — `question` (5–2000 chars), optional `doc_id` (scope to one document), optional `top_k` (1–20), `include_graph_evidence` (bool, default true), optional `graph_depth` (1–3).
**Output:** `AnswerResponse` — 14 fields including `answer`, `answer_type`, `evidence_quality`, `confidence_note`, `citations`, `retrieved_chunks`, `graph_evidence`, `confidence`, `limitations`, `latency_ms`.
**Why it matters:** The core capability. The full trust layer is returned in a single structured response, enabling downstream applications to make informed decisions about how to present or act on the answer.

---

### `GET /api/v1/documents`
**Purpose:** List all documents currently indexed in the vector store.
**Input:** None.
**Output:** `ListDocumentsResponse` — `documents: list[DocumentSummary]`, `total: int`. Each summary has `doc_id`, `doc_title`, `source_url`, `chunk_count`.
**Why it matters:** Allows a frontend or API consumer to enumerate the corpus without querying the vector store directly.

---

### `GET /api/v1/documents/{doc_id}`
**Purpose:** Retrieve detailed metadata for a single indexed document.
**Input:** `doc_id` (path parameter).
**Output:** `DocumentDetail` — `doc_id`, `doc_title`, `source_url`, `page_count`, `chunk_count`, `sections` (list of detected headings), `status`.
**Why it matters:** Supports document management workflows and debugging of the ingestion pipeline.

---

### `DELETE /api/v1/documents/{doc_id}`
**Purpose:** Remove a document and all its chunks from the vector store.
**Input:** `doc_id` (path parameter).
**Output:** `DeleteDocumentResponse` — `doc_id`, `deleted: bool`, `message`.
**Why it matters:** Enables corpus management. Note: graph entities from a deleted document are not removed (entities may be shared across documents) — this is documented behaviour.

---

### `GET /api/v1/graph/stats`
**Purpose:** Return knowledge graph statistics.
**Input:** None.
**Output:** `GraphStatsResponse` — `total_entities`, `total_relations`, `graph_provider`, `graph_enabled`.
**Why it matters:** Confirms that entity extraction ran during ingestion and that the graph layer is populated. Useful for diagnosing whether spaCy is installed.

---

### `GET /api/v1/graph/neighbours`
**Purpose:** Retrieve entities within N hops of a named entity and all direct edges from that entity.
**Input:** `entity` (query param), `depth` (1–3, query param).
**Output:** `GraphNeighboursResponse` — `entity`, `neighbours: list[GraphNode]`, `edges: list[GraphEdge]`, `total_entities`, `total_relations`.
**Why it matters:** Exposes the knowledge graph for interactive exploration. A frontend graph visualisation would call this endpoint to render force-directed views of entity relationships.

## 13. Security and Responsible AI

**Security practices:**

- **No hardcoded secrets.** API keys (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `NEO4J_PASSWORD`) are read exclusively from environment variables via pydantic-settings. All have empty string defaults and are excluded from `repr()` output.
- **`.env` is gitignored.** The `.env` file is listed in `.gitignore`. `.env.example` is committed as a template with all values empty or set to safe defaults (mock mode).
- **No private datasets committed.** The repository ships one small synthetic sample policy text (`sample_data/sample_policy.txt`). No real documents, PDFs, or proprietary datasets are included. The `document_loader.py` docstring directs users to public sources.
- **File-size caps.** URL ingestion enforces a configurable `MAX_DOCUMENT_SIZE_MB` limit (default 50 MB) with streaming to avoid memory exhaustion.
- **Input validation at the boundary.** All user input is validated by Pydantic v2 before reaching the pipeline. Question length (5–2000 chars), `top_k` range (1–20), and `graph_depth` range (1–3) are enforced at schema level.

**Responsible AI practices:**

- **Citation-first answer generation.** The system prompt mandates inline page citations. Every API response includes a structured `citations` list linking claims to source passages.
- **Trust gate against hallucination.** The pipeline never calls the LLM when no passages are retrieved. This eliminates the primary hallucination pathway (generating content from empty context).
- **Confidence and limitations always present.** `confidence`, `evidence_quality`, `confidence_note`, and `limitations` are returned in every response. When confidence is None (no corpus), this is made explicit rather than returning a zero score.
- **Deterministic `answer_type` classification.** `cited`, `partial`, `refused`, and `no_corpus` are assigned by pattern-matching the answer text and inspecting the pipeline state — not by asking the LLM to self-classify.
- **Mock mode for safe demos.** `LLM_PROVIDER=mock` produces structured, context-aware stub answers with no API calls, no costs, and no external data transmission. This is the default for CI and Codespaces.
- **Responsible AI documentation.** `docs/responsible_ai.md` documents intended use, out-of-scope use cases, eight named trust mechanisms, and operator obligations.

## 17. Current Limitations

These are real, known limitations stated honestly:

1. **Mock provider does not produce real intelligence.** `MockProvider` returns a structured stub that confirms the pipeline is working but does not generate useful analytical answers. Real LLM output requires configuring `LLM_PROVIDER=anthropic` or `LLM_PROVIDER=openai` with a valid API key.

2. **Real embeddings require a model download.** The first time the server starts, `sentence-transformers/all-MiniLM-L6-v2` (~90 MB) is downloaded from HuggingFace Hub. Subsequent starts use the cached model. CI caches the model between runs.

3. **Graph extraction quality depends on spaCy.** When `spaCy` and `en_core_web_sm` are installed, NER extracts nine entity types. Without them, the entity extraction step is silently skipped, the graph remains empty, and `graph_evidence` is always an empty list. The `limitations` field explicitly notifies callers when this is the case.

4. **Relation extraction is rule-based.** The current relation extractor uses proximity co-occurrence rules rather than dependency parsing or a trained relation extraction model. It produces typed triples (MENTIONED_WITH, OPERATES_IN, TARGETS_BY, GOVERNED_BY) but misses complex semantic relationships.

5. **No frontend implemented.** The frontend is planned as documented in `frontend/README.md`. The backend API is complete and ready for integration, but there is no browser interface today.

6. **Evaluation uses synthetic documents.** The evaluation framework in `test_evaluation.py` and `scripts/evaluate.py` runs against four synthetic policy documents. There is no human-labelled ground-truth benchmark dataset. Evaluation results measure pipeline structure and observable properties, not real-world retrieval quality.

7. **ChromaDB not installed by default in minimal environments.** If only core dependencies are installed without the full `pip install -e '.[dev]'`, ChromaDB may be absent. The server falls back gracefully: set `VECTOR_STORE_PROVIDER=memory` to use the in-memory store.

8. **No Docker image or Compose file.** Containerised deployment is not included in this version. The project runs locally or in Codespaces but requires manual dependency installation for deployment to a server.

9. **No authentication.** The API has no authentication layer. All endpoints are publicly accessible. Adding token-based authentication (FastAPI's OAuth2, API keys, or JWT) is a straightforward extension.

10. **Confidence scoring is heuristic.** The confidence formula (mean top-3 cosine similarity + graph bonus) is a proxy for answer reliability, not a calibrated probability. It should be treated as an ordering signal, not an absolute measure of correctness.

## 18. Future Roadmap

Practical next steps, ordered roughly by value-to-effort ratio:

1. **Frontend dashboard.** A Streamlit prototype could be built in a few hours to provide document upload, query input, and citation display. A React + TypeScript application would be production-quality and could visualise graph evidence.

2. **Interactive graph visualisation.** The `GET /api/v1/graph/neighbours` endpoint already exposes graph data. A force-directed D3.js or Cytoscape component in the frontend would let users explore entity relationships interactively.

3. **Hosted demo.** Deploy to Railway, Render, or Hugging Face Spaces with `LLM_PROVIDER=mock` and `VECTOR_STORE_PROVIDER=memory` for a live, no-API-key-required demo that reviewers can access from a URL.

4. **Docker / Docker Compose.** A `Dockerfile` and `docker-compose.yml` would make local setup reproducible across all platforms and enable straightforward cloud deployment.

5. **Authentication.** FastAPI's dependency injection makes it straightforward to add API key authentication or OAuth2 without modifying existing route handlers.

6. **Real benchmark dataset.** Using a publicly licensed policy document collection (EUR-Lex, World Bank Open Knowledge, UN Document Portal) with human-annotated question-answer pairs would replace synthetic evaluation with real retrieval quality measurement.

7. **Advanced entity/relation extraction.** Replacing the rule-based relation extractor with a dependency-parse-based extractor or a fine-tuned relation extraction model would improve graph quality significantly.

8. **Improved confidence calibration.** Replacing the heuristic confidence formula with a calibrated retrieval scorer (e.g., BM25 fusion, cross-encoder reranking) would make the confidence signal more reliable.

9. **Observability.** Adding request-level tracing (OpenTelemetry), Prometheus metrics, and a Grafana dashboard would make the system production-observable.

10. **Streaming responses.** FastAPI supports `StreamingResponse`; adding streaming for the answer generation step would improve perceived latency for long answers.
